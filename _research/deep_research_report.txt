Deep Research Report: AI‑Enhanced Autofill for
yadasa/exempliphai
Executive summary
The yadasa/exempliphai repository is a Manifest V3 Chrome extension whose autofill system is already a
strong hybrid of deterministic DOM/heuristic filling and AI assistance. The current architecture has
three key pillars:
1. Extension popup UI (Vue 3 + Vite) for maintaining a user profile (stored in
chrome.storage.sync ) and resume artifacts/details (stored in chrome.storage.local ).

1

2

2. Content scripts that detect application pages, match fields using a platform-keyed mapping table
plus fuzzy matching, and write values into controls while emitting DOM events so frameworks (React,
4
etc.) react correctly. 3
3. A Gemini API integration used in two places: (a) resume parsing into structured JSON during file
upload in the popup, and (b) generating “AI answers” for user-selected fields in job applications. 5
6

What is not yet fully covered (or not covered consistently) is a generalized, AI-driven approach that can
robustly handle all control types on arbitrary sites—especially contenteditable, rich text editors, highly
custom widget ecosystems (React Select variants, MUI/AntD controls, bespoke ATS widgets), and date/
time pickers. The repo also uses broad match patterns in manifest.json and therefore must be
unusually careful about where AI calls and autofill triggers occur.

7

This report provides:
• A complete inventory (from the repo scan) of form-related files/components/handlers as a
table.
• A field-type-by-field-type set of AI parsing/mapping strategies with prompt templates, edge cases,
and concrete heuristics.
• A detailed integration design with Mermaid flowcharts and an ER model.
• A security/privacy/consent plan aligned to Chrome extension security constraints, GDPR, and
CCPA/CPRA.
• A prioritized implementation plan with estimates, pseudocode, and test/CI guidance.
• Performance and cost scenarios grounded in official pricing for Gemini and OpenAI, and
standards-based form semantics.

Key codebase takeaways relevant to AI autofill
• The DOM filling side already uses several standards-aligned signals like label , aria-* , and
other attributes; aligning it systematically with accessible name computation significantly
improves accuracy on modern UIs. 44
• The extension’s content scripts run in an isolated world (important for security boundaries), but AI
calls from content scripts still require careful consent and data minimization because they can be
triggered on arbitrary matching pages. 45
• File uploads are supported via assigning files rather than setting the file input’s value .
Browser security restrictions make value assignment ineffective, so file handling must remain
DataTransfer-/FileList-based.

AI-based parsing and mapping strategies by form control type
This section proposes how to evolve from the current “mapping table + fuzzy match” to an AI-assisted form
understanding engine that can reliably fill text, textarea, radio, checkbox, select, date/time, file inputs,
contenteditable, and custom widgets.

Assumptions about profile schema
Because the profile schema is unspecified, assume the following flexible model:
• Source of truth remains the existing key/value store: chrome.storage.sync (flat keys like “First
Name”, “Expected Salary”) plus chrome.storage.local structured objects (resume details).

31

• Introduce a canonical internal schema (optional) while maintaining backward compatibility: e.g.,
person.givenName , contact.email , employment.workAuth.usAuthorized , etc. Canonical
schema improves AI mapping and normalization, but the system must still operate purely off the
existing keys.

Cross-cutting strategy: “semantic field descriptor” + “fill plan”
Instead of asking the model to “fill the page,” ask it to produce a FillPlan for a controlled list of fields:
1. Extract field descriptors (deterministic):
2. Primary: accessible name (label/aria) and role. 47
3. Secondary: name , id , placeholder , autocomplete , inputmode , pattern , required ,
DOM neighbors (section headings), and for option-based controls: enumerated options (limited).
4. Run deterministic mapping first using:
5. HTML label associations and conventions. 48
6. ATS-specific map table (your existing fields maps).

4

7. Use AI only for the remaining unresolved or ambiguous fields, and constrain AI output to a JSON
schema (structured outputs / strict JSON). 49

Text inputs
Best targets: type=text , type=email , type=tel , type=url , plus unlabeled text inputs.
Deterministic heuristics - Use label association logic per HTML spec: <label for> or wrapping label.
50

- Prefer semantically meaningful hints: -

autocomplete=email ,

autocomplete=given-name ,

autocomplete=family-name , etc. - type=email for Email - type=tel for Phone - Normalize user
profile values: - Derive first/last from full name - Normalize phone formatting (E.164 fallback when required)
- URL normalization ( https:// prefix when missing)
AI mapping (when needed) - Use a low-latency model to pick the best profile attribute and apply
transformations. - Ask for: value , source_key , transform , confidence , reason .
Edge cases - Multiple “Address” fields: “Email Address” vs street address (already noted as a bug class in
repo docs). 31

4

- “Preferred name,” “Legal name,” “Name pronunciation” should not map to “Full Name” blindly. - “LinkedIn
profile” sometimes expects username rather than URL.

Textareas
Two distinct categories 1. Short textareas (notes, brief explanations): treat like text inputs. 2. Long-form
questions (motivation, experience narratives): AI generation is appropriate.
Deterministic heuristics - If textarea is “long” (by rendered height or maxlength), treat as narrative. Extract question text via accessible name algorithm and nearby prompt text (carefully bounded). 44
AI generation strategy - Input: question, (optional) job title/company, stored resume details, and only the
minimum relevant profile fields. - Output: a concise answer under a budget, optionally with ATS-specific
guidance (your code already applies site constraints). 3
Edge cases - Textarea is part of a rich editor wrapper (Quill/Draft/etc.)—may not respond to .value ; treat
it as “custom widget” or “contenteditable” path. - “Cover letter” vs “additional information” vs “why do you
want to work here” should produce different tone and length.

Radio buttons
Deterministic heuristics - Group by shared

name . - Compute each option’s label text (label[for],

wrapping labels, aria-label). This aligns with HTML labeling and accessible naming concepts.
- Match against normalized user value (Yes/No synonyms; domain-specific overrides).

51

AI mapping - Only needed when the question is nuanced (e.g., “Will you require sponsorship now or in the
future?”) or when Yes/No is not clearly derivable. - Provide the option list to AI and ask it to choose exactly
one option id/value.
Edge cases - “Legally authorized to work” depends on geography; AI should not guess—should follow
user’s stored setting or request confirmation. - Radios implemented as ARIA listbox options rather than
<input type=radio> .

Checkboxes
Checkboxes are tricky because they can be: - A single consent (boolean) - A Yes/No pair implemented as
multiple checkboxes - A true multi-select list
Deterministic heuristics - If a checkbox has a clear “I agree” label, default behavior should be never autocheck without explicit user opt-in. - For multi-checkbox question groups, treat like radios but allow
multiple checked values.
AI mapping - Provide group context and ask AI for a set of options to check, each with confidence.
Edge cases - Required “acknowledgement” checkboxes: treat as consent gates; require human action.

5

Native selects
Native <select> is amenable to deterministic matching: - Enumerate options and apply fuzzy matching +
synonyms. - Handle “country” mapping (ISO alpha-2) and common location strings (existing code already
does a version of this). 3
AI helps when: - Options are verbose or oddly phrased (EEO disclosures) - Multiple similar options exist
(“Decline”, “Prefer not to say”, “Self describe”)
Edge cases - Selecting placeholders (“Select…”) must be avoided. - Some selects require a preceding click to
open.

Date/time pickers
There are three common patterns: 1. Native inputs: type=date , type=time , type=datetime-local
2. Split fields: month/day/year inputs (Workday-style) 3. Fully custom pickers (React/MUI)
Deterministic strategy - Detect the pattern by DOM: - input.type for native. - clusters by aria/labels
(“Month”, “Day”, “Year”). - Normalize data: - Store canonical date values internally as ISO-8601, then format
for the control. - For split fields, fill parts deterministically (month number vs name).
AI mapping - Only for ambiguous date fields (“Available start date”, “Graduation date”) to decide which
profile attribute applies and what default should be if missing.

File inputs
Browsers block setting a file picker value by script; file interactions must use supported mechanisms
(FileList/DataTransfer), and reading files must come from explicit user selection.

52

Deterministic strategy - Classify which document is expected (Resume/CV/Cover Letter/Transcript/Portfolio
PDF) using: - label text, aria-label, surrounding prompt - accept types ( accept=".pdf" etc.) - Never
upload Resume into Cover Letter input unless the user explicitly enables this mapping (the repo already
flags this as a known hazard). 31
AI mapping - Allow AI to classify “document intent” from the prompt text, but do not let AI decide to upload
a document without meeting consent rules.

Contenteditable areas
Contenteditable is common in rich editors. It does not use .value . You must use DOM Range / Selection
and dispatch beforeinput / input events for framework-integrated editors.
Deterministic strategy - Detect: el.isContentEditable or [contenteditable="true"] . - Extract
label via accessible name computation and nearby prompt text. 53
- Set content: - Prefer textContent for plain editors; innerHTML only if formatting is required and safe.
- Dispatch InputEvent('input') plus Event('change') if needed.

Edge cases - Editor is an iframe (common in enterprise apps). Content scripts may need all_frames
(already enabled) but still must locate editable frame DOM carefully.

Integration design for AI autofill
This design builds on existing triggers (Autofill Now button and message-based AI answer generation) and
adds a single “FillPlan” orchestration layer.

Where to call AI APIs
Recommendation: Use a tiered approach.
• Tier 0 (no AI): Deterministic mapping (existing fields + label/aria/attribute matching) for highconfidence fields. 4
• Tier 1 (small AI call): “Map these unresolved fields to profile keys” with strictly structured output.
49

• Tier 2 (generation): Only for long-form narrative answers (textarea/contenteditable), using resume
details and question prompt.

Client-side vs server-side processing
Option A: Client-side (BYOK, current pattern) - Pros: no backend, user owns key, easiest to ship. - Cons: AI
key stored locally; network calls from extension contexts must be carefully permissioned and consented.

7

Option B: Server-side proxy - Pros: central key management, consistent governance, easier caching/
batching, supports enterprise policies. - Cons: introduces a data processor role; stronger GDPR/CCPA
requirements and security obligations.
Given the repo already stores an API key and calls Gemini directly, an incremental plan is:
1. Implement the new FillPlan engine locally (Gemini BYOK).
2. Add an optional “Use hosted service” mode later.

Required endpoints (if you add a backend)
If you implement Option B, minimally:
• POST /v1/autofill/plan → returns FillPlan JSON for a form snapshot.
• POST /v1/autofill/generate → returns generated text for a single prompt field.
• POST /v1/resume/parse → returns structured resume details from an uploaded PDF (or accept
already-extracted text).

Latency, batching, and caching strategies
Batching - Batch unresolved fields into one AI call per page (or per “step” in multi-step flows) instead of perfield calls. - For long-form text generation, call per-field, but only when user requests or when confident it’s
required.
Caching - Cache stable prompts (e.g., mapping templates and recurring ATS field shapes) using: - Gemini
caching mechanisms when available (pricing indicates caching exists for Gemini 3 Flash Preview). 55
- OpenAI cached input pricing incentives (if used). 56

9

Time budgets - Tier 0 fill should complete in <200ms. - Tier 1 mapping call target: 500ms–2s depending on
model and payload size. - Tier 2 generation: 1–5s; must be accompanied by UI feedback and cancel/undo.

Security, privacy, and consent requirements
Chrome extension security realities
• Content scripts run in isolated worlds, reducing direct DOM-script leakage risk, but they still interact
with potentially hostile pages. 57
• The extension uses chrome.storage (sync and local). Chrome notes quotas and sensitivity
concerns, and recommends caution with sensitive data in sync .

58

• MV3 permission discipline matters; Chrome recommends using optional permissions where feasible
and avoiding “future-proof” permission grabs. 59

Data minimization rules for AI requests
Adopt a policy-driven “least disclosure” model:
• Never send entire chrome.storage.sync by default (the current AI-answer path appears to
include large context). Instead:
• For mapping: send only the candidate keys and values relevant to the unresolved fields (or send keys
+ hashed values when possible).
• For generation: send resume details and only the handful of fields relevant to that prompt.
• Do not send user documents (resume PDF) except in an explicit “parse resume” workflow initiated
from the user’s upload action (already aligned with FileReader’s “user-selected files only” principle).
60

Encryption, key handling, and logging
API key storage - Prefer storing AI keys in chrome.storage.local or chrome.storage.session
rather than sync (sync replicates across devices and is quota-limited). 58
- Consider encrypting secrets using WebCrypto with a user-provided passphrase (optional, UX tradeoff).
Audit logs - Store audit events locally; avoid storing raw sensitive values: - Keep old/new values only as
hashes unless user opts in. - Keep “field fingerprint” derived from stable attributes (name/id + label hash) to
support debugging without full DOM capture.

Consent UX
Implement explicit consent layers:
• Global toggle: “Enable AI-assisted autofill” (off by default).
• Per-site toggle: “Allow AI for this domain” and “Allow sending resume details to AI.”
• Per-action review: for low-confidence or sensitive fields (EEO, disability, veteran, sponsorship),
require confirmation.

GDPR and CCPA/CPRA considerations
If you operate a backend or otherwise determine processing purposes, you may become a controller/
processor under GDPR and must align to:
• Security and integrity/confidentiality principles and lawful basis constraints.
• Data subject rights such as deletion/erasure. 62

61

For CCPA/CPRA, users may have rights to know, delete, correct, and limit use of sensitive personal
information depending on whether you meet applicability thresholds and roles. 63
Practical engineering implications: - Provide data export (already present) and add “delete all data” + “delete
AI logs.” 14
- Provide transparent notices in the UI explaining what is sent to AI, when, and why.

Fill executor with undo + confidence thresholds

// contentScripts/fillExecutor.js
export function applyFillPlan(fillPlan, { minConfidence = 0.75 }) {
const undoStack = [];
for (const action of fillPlan.actions) {
if (action.confidence < minConfidence && !action.userApproved) continue;
const el = findElementByFingerprint(action.fieldFingerprint);
if (!el) continue;
const oldValue = readCurrentValue(el);
const ok = setControlValue(el, action); // routes to type handlers
if (ok) {
undoStack.push({ fieldFingerprint: action.fieldFingerprint, oldValue });
logAuditEvent(action, oldValue);
}
}
return { undoStack };
}

and add a single “Undo Autofill” button. - < 0.75: require review overlay before applying.
Review overlay features: - Show field label, chosen value, and “why” snippet. - Allow user to switch among
top-3 suggestions (AI returns alternatives). - Provide “never fill this field on this site” rules.

Undo and audit logs
Undo mechanism: - Keep an in-memory undo stack for the current run. - Persist last run metadata (not
values) to local storage for debugging.
Audit logs: - Store timestamps, domain, field fingerprint, confidence, and hashed old/new values.
